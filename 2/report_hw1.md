# Отчёт к ДЗ №1
## 0. Архитектурное ката
### Tales Of A Fourth Grade
An elementary school system comprehensive student management system

**Requirements:**
+ Track absences;
+ Tardies and excuses (entered by parents, faculty or staff);
+ Generate reports on student activities;
+ Be accessible from the playground;
+ Track student grades and assignments (completed and due);
+ Parent-teacher forums;
+ Run as an SaaS system from a hosting center.

**Users:**
faculty, staff and student parents

## 1. Отсев
На данном этапе выполнено несколько задач:
### Отсев возможных функций
|Функция|Причина отсева|
|--------------|--------------|
|Наличие мобильного приложения|Не указано чётко в требованиях. Разработка потребует значительно большего количества версии, чем только для веб-версии, при этом ценность конечного продукта не возрастёт кратно.|
|Интеграция с внешними сервисами|Отсутствует в исходных требованиях. Создание и поддержка API повлекут за собой неоправданно высокие расходы для MVP.|
|Обеспечение аудио- и видеосвязи в форуме|Функционал отсутствует в исходных требованиях, разработка потребует значительных ресурсов и будет неоправданно дорога для MVP.|
|Управление другими элементами образовательного процесса (питание, читательский билет и пр.)|Не указано в требованиях, вызовет дополнительные неоправданные для MVP затраты ресурсов и времени.|
|Наличие инструментов ИИ|Функционал отсутствует в исходных требованиях. Разработка / Интеграция с внешними решениями и поддержание API вызовут дополнительные временные и финансовые затраты, а потому могут быть реализованы после прямого указания со стороны заказчиков.|
|Работа в офлайн-режиме|Не отражено в исходных требованиях. Разраюботка потребует значительных затрат ресурсов, поэтому данный функционал не предусмотрен в MVP.|
|Отдельный функционал для учащихся|Не прописан в требованиях. Разработка отдельного формата ЛК и всех необходимых дополнений потребует существенных временных и финансовых затрат, что не требуется на этапе создания MVP.|
|Возможность ведения личной переписки между родителями и сотрудниками|Внедрение дополнительного функционала повлечёт сложность в реализации, но не придаст продукту значительно бОльшую ценность (родители могут общаться с преподавателями напрямую по телефону или во внешних мессенджерах).|

### Матрица рисков
|Риск|Мера по смягчению|
|--------------|--------------|
|Неоднозначность формулировки *be accessible from the playground*: от разработчика могут ожидать мобильное приложение или офлайн-работу.|Чётко прописанные критерии: веб-интерфейс, достаточная скорость интернета для загрузки.|
|Непонятны требования к форуму между родителями и преподавателями.|Необходимо указать однозначные критерии приёма.|
|Не предумострена пиковая нагрузка в утренние часы.|Необходимо изначально проектировать систему с запасом нагрузки.|

### Критерии приёма
1. Загрузка веб-версии приложения занимает не более 10с при стабильной скорости интернета от 2 Мбит/с.
2. Загрузка отдельных страниц уже открытого приложения занимает не более 5с при стабильной скорости интернета от 2 Мбит/с.
3. Отметка отсутствия / присутствия ученика (со стороны преподавателей) и указание причины отсутствия ученика (со стороны родителей) выполняются не более чем в 3 действия, одно из которых -- внесение в систему развёрнутого комментария (прр необходжимости).
4. Отчёты по успеваемости и занятиям учеников выгружаются в течение 15с при стабильной скорости интернета от 2 Мбит/с.
5. Форум для преподавателей и родителей функционирует в онлайн-режиме, доступна загрузка файлов в фото и видеофайлов.
6. Преподавателям и родителям доступны данные только их учеников / детей.
7. Интерфейс приложения интуитивно понятен и прост.

## 2. Scope Refinement
На этом этапе уточняются и детализируются гранницы проекта.
### Типы пользователей
+ Сотрудники подразделения
+ Преподаватели
+ Родители
### Ключевой функционал
+ **Наличие карточек учеников и работа с ними:** в данной карточке отдельно хранится информация о посещаемости, оценках, заданиях учащихся. Кроме того, к карточкам прикрепляются все необходимые документы.
+ **Отслеживание отсутствий и опозданий учеников:** электронный журнал с датами, учениками и отметками посещаемости (значения из списка: "Присутствует", "Отсутствует", "Отсутствует по уважительной причине").
+ **Подача заявлений об отсутствии учеников по уважительным причинам:** загрузка доступна в ЛК родителей, а также может быть реализована в ЛК преподавателей и сотрудников (они могут вносить информации в карточки учащихся). Ссылки на документы также доступны в электронном журнале при нажатии на кнопку "Отсутствует по уважительной причине" напротив ФИО конкретного ученика.
+ **Создание отчётов по конкретному учащемуся:** выгрузка доступна пользователям через карточку ученика. Возможен экспорт в форматы xlsx и pdf.
+ **Доступ к отслеживанию успеваемости и заданий учащихся:** в карточках учащихся отмечаются предметы, даты домашних заданий, их названия и описания, а также статусы ("Не сдано", "На проверке", "Проверено") и отметки за проверенные задания.
+ **Доступность приложения в школе, дома и на улице:** веб-приложение может быть открыто в любом месте.
+ **Форумы для преподавателей и родителей:** средство для общения между преподавателями и учениками. Доступно в онлайн-режиме, есть возможности отправки текстовых сообщений, а также вложений в виде документов и изображений (расширения png, jpg, jpeg, pdf, doc, docx, rtf, txt). Возможно получение уведомлений о получении сообщений.
+ **Запуск приложения как SaaS:** каждая школа является отдельной единицей в общей системе школ, данные о пользователях системы и учениках изолированы от данных других школ. Помимо этого, реализованы централизованное развёртывание сервисов, управление версиями и мониторинг из одного ЦОДа.

## 3. Функциональные требования
### Управление ролями и пользователями
+ Регистрация и дальнейшая авторизация с учётом ролей и прав доступа (администратор, сотрудник (в т. ч. преподаватель), родитель / законный представитель учащегося).
+ Администратор имеет право назначать роли и управлять правами доступа пользователей, а также обновляет списки учащихся (добавляет новых и удаляет выбывших) и делит их на группы (классы, параллели, группы для курсов по выбору).
+ Родители и преподаватели имеют доступ к карточкам только своих детей / учеников соответственно.

### Посещаемость
+ Преподаватели и администраторы имеют право вносить данные о посещаемости занятий учениками в электронный журнал, родители могут только просматривать их в общем журнале или личном кабинете учащегося.
+ Сотрудники и родители имеют право вносить документы, подтверждающие уважительную причину пропуска, в личные карточки обучающихся. Документы должны быть доступны к просмотру как в личных карточках учеников, так и в таблицах с успеваемостью (по ссылке на личную карточку учащегося).

### Оценки и задания
+ Преподаватели создают задания в своём личном кабинете с указанием полей: название, дата выдачи, дата сдачи, текст задания, максимальный балл.
+ Преподаватели имеют право назначать задания как для групп учащихся, так и для отдельных учеников.
+ Родители могут видеть данные задания в карточке учеников. Также они видят статус задания у своего ребёнка: "Не сдано", "На проверке", "Проверено".
+ Оценка, рассчитанная по формуле с учётом весов элементов контроля, отображается в личной карточек обучающегося.

### Отчёты
+ Преподаватели и родители могут скачать отчёт по ученику из его / её личной карточки. В отчёте по выбору могут быть отражены посещаемость, успеваемость, список текущих и просроченных заданий. Возможно регулировать срок, за который оформляется отчёт.
+ Экспорт отчёта возможен в форматах pdf и xlsx.
+ Сотрудники и администрация имеют возможность формирования и просмотра агрегированного отчёта по всем учащимся и группам, доступ к карточкам которых они имеют.

### Форум для учителей и родителей
+ Форум работает при наличии сети.
+ Возможно создание форума для каждой из групп, созданных администратором, с добавлением прикреплённых к этой группе сотрудников, представителей администрации и родителей.
+ В форумах есть возможность отправки текстовых сообщений, а также документов (pdf, doc, docx, rtf, txt) и изображений (png, jpg, jpeg).
+ Есть возможность настроить уведомления о получении сообщений в форуме (аналогично можно отписаться от их получения позднее).

### SaaS-администрирование
+ Суперадминистратор управляет клиентами: ему доступны функции создания, изменения, удаления клиентов
+ Клиенты и их данные хранятся изолированно друг от друга: пользователи видят данные только об обучающихся / сотрудниках, с которыми они связаны вследствие осуществления образовательного процесса.

## 4. Нефункциональные требования
**1. Доступность приложения**
+ Загрузка веб-версии приложения занимает не более 10с при стабильной скорости интернета от 2 Мбит/с.
+ Загрузка отдельных страниц уже открытого приложения занимает не более 5с при стабильной скорости интернета от 2 Мбит/с.
+ Отчёты по успеваемости и занятиям учеников выгружаются в течение 15с при стабильной скорости интернета от 2 Мбит/с.
+ Загрузка и отправка сообщений и файлов в форуме выполняется не более чем за 3 с при стабильной скорости интернета от 2 Мбит/с.
+ Доступность > 99% в учебное время. Технические перерывы запланированы на неучебное (преимущественно, ночное) время.
+ Показатели доступности должны сохраняться в часы пиковой нагрузки (время начала уроков, когда отмечается посещаемость; время окончания уроков, когда выставляются оценки, и пр.).

**2. Удобство использования (Usability)**
+ Авторизация выполняется в три действия: ввод логина, ввод пароля, нажатие на кнопку "Вход". В дальнейшем возможна реализация 2ФА.
+ Учёт успеваемости и посещаемости должен производиться в 2 действия: нажатие на ФИО ученика в списке класса --> выбор значения из выпадающего списка (для посещаемости: "Присутствует", "Отсутствует", "Отсутствует по уважительной причине"; для успеваемости: "1", "2", "3", "4", "5", "Зачёт", "Незачёт").
+ Сообщения в форуме отправляются в одно касание, файлы прикрепляются в два действия (нажатие на кнопку для прикрепления файлов --> выбор файла из проводника / галереи).
+ Отчёт выгружается нажатием на одну кнопку: "Скачать отчёт". Дополнительно можно выбрать значения кнопок "Выбор даты" (возможность выбора диапазона при помощи календаря), "Необходимые данные" ("Успеваемость", "Посещаемость"), "Предмет" (выбор дисциплины, по которой будет сформирован отчёт). По умолчанию загружается отчёт с данными и по успеваемости, и по посещаемости по всем предметам за последние 30 дней.

**3. Адаптивность**
+ Формат отображения личного кабинета настраивается автоматически в соответствии с характеристиками экрана устройства (размер, разрешение, яркость).
+ Возможно использование с мобильных устройств (Android, iOS, HarmonyOS), ноутбуков и ПК (Windows, MacOS, Linux).
+ Возможен выбор размера шрифта (мелкий, обычный, крупный), по умолчанию настроен крупный.
+ Доступная тема по умолчанию -- светлая.

**4. Производительность**
+ 95% запросов к серверу выполняются быстрее чем за 400 мс.
+ Пиковая нагрузка -- не менее 100 записей о посещаемости / успеваемости в секунду, 200 сообщений в форуме, 100 загруженных отчётов.

**5. Безопасность данных**
+ Все соединения осуществляются через HTTPS.
+ Авторизация выполняется через защищённый токен.
+ Клиенты и их данные хранятся изолированно друг от друга: пользователи видят данные только об обучающихся / сотрудниках, с которыми они связаны вследствие осуществления образовательного процесса. Изоляция реализована на уровне приложения и БД.
+ Пароли хэшируются.
+ Доступ журнал действий (в частности, отражаются все входы в систему).

**6. Масштабируемость**
+ Архитектура должна предоставлять возможность горизонтального масштабирования (к примеру, для включения новых клиентов (школ) в систему).
+ База данных -- с асинхронной репликацией данных (обновление раз в день ночью).

**7. Локализация**
+ Поддержка русскоязычного интерфейса.
